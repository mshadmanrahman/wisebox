## 1. System Overview

Wisebox is a three-portal digital property management platform for the South Asian diaspora. The rebuild produces three connected applications sharing a single API backend.

### Portal 1: Customer Frontend (Next.js)
Where users sign up, create properties, upload documents, select services, pay via Stripe, and book consultations.

### Portal 2: Admin Dashboard (Laravel Filament)
Where Wisebox admins manage property types, ownership statuses, document types, services, orders, tickets, consultants, and dynamic form configurations.

### Portal 3: Consultant Portal (Laravel Filament, separate panel)
Where assigned consultants view their tickets, review client documents, prepare for meetings, and deliver service outcomes.

### Shared: Marketing Site (Next.js)
Static/SSR marketing pages, free assessment tool, lead generation. Same Next.js app as the customer frontend, using route groups.

## 2. Technology Stack

### Backend
| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Framework | **Laravel 11** (PHP 8.3+) | Same as existing; team knows it |
| Admin Panel | **Laravel Filament 3** | Replaces custom Blade admin; rapid CRUD, forms, tables |
| API Auth | **Laravel Sanctum** | Token-based SPA auth; session auth for admin |
| Database | **MySQL 8** | Same as existing |
| File Storage | **AWS S3** | Document uploads (deeds, khatians, etc.) |
| Queue | **Laravel Queue (Redis driver)** | Async jobs: emails, Stripe webhooks, ticket creation |
| Cache | **Redis** | Session store, location data cache, rate limiting |
| Search | **Laravel Scout + Meilisearch** (optional) | Property/service search |

### Frontend (Customer Portal + Marketing)
| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Framework | **Next.js 14** (App Router) | SSR for marketing, SPA for portal |
| Language | **TypeScript** | Type safety across the frontend |
| State | **Zustand** | Lightweight global state (auth, form drafts) |
| Forms | **React Hook Form + Zod** | Complex multi-step property forms with validation |
| API Client | **TanStack Query (React Query)** | Caching, optimistic updates, auto-refetch |
| Styling | **Tailwind CSS + shadcn/ui** | Rapid UI with consistent design system |
| File Upload | **react-dropzone + tus-js-client** | Resumable uploads for large documents |
| Payments | **@stripe/stripe-js + @stripe/react-stripe-js** | Stripe Elements for checkout |

### Infrastructure
| Layer | Technology |
|-------|-----------|
| Frontend Hosting | **Vercel** |
| Backend Hosting | **AWS EC2 / Laravel Forge** |
| Database | **AWS RDS (MySQL)** |
| File Storage | **AWS S3** |
| CDN | **CloudFront** (for S3 assets) |
| DNS | **Cloudflare** (mywisebox.com) |
| CI/CD | **GitHub Actions** |
| Monitoring | **Laravel Telescope** (dev) + **Sentry** (prod) |

### Third-Party Integrations
| Service | Purpose | SDK/API |
|---------|---------|---------|
| **Stripe** | Payment processing | Stripe PHP SDK + Stripe.js |

### Request Flow

```
Customer Browser
  → Next.js (Vercel) serves SSR/static pages
  → Client-side JS calls Laravel API via Sanctum tokens
  → Laravel processes request, talks to MySQL/S3/Redis
  → Returns JSON response
  → Next.js renders UI update

Admin/Consultant Browser
  → Direct to Laravel server
  → Filament renders admin/consultant panels
  → Session-based auth (standard Laravel)
```

```php
class StripeWebhookController {
    public function handle(Request $request) {
        $event = Webhook::constructEvent(
            $request->getContent(),
            $request->header('Stripe-Signature'),
            config('services.stripe.webhook_secret')
        );

        match ($event->type) {
            'checkout.session.completed' => $this->handleCheckoutComplete($event),
            'payment_intent.payment_failed' => $this->handlePaymentFailed($event),
            'charge.refunded' => $this->handleRefund($event),
        };
    }
}
```

## Appendix B: Key Design Decisions (from meetings)

1. **2-step accordion form** (not 7-step wizard) for property registration
2. **Human-readable ownership labels** ("I purchased it" not "Ownership by Purchase")
3. **White-label architecture**: JSON-driven forms for future multi-country support
4. **Auto-save drafts** every 30 seconds
5. **80% document completion** triggers basic service eligibility
6. **Color-coded status**: Green (>=80%), Yellow (40-79%), Red (<40%)
7. **Dual login**: Google OAuth + email/password with OTP
8. **5-step progressive signup**: Name > Password > Country > ToS > OTP
9. **Three equal-weight CTAs** on dashboard: Add property, Talk to expert, Get assessment
10. **"Feeling lost?"** messaging for empty state
11. **International positioning**: Not Bangladesh-only, but Bangladesh first
12. **WhatsApp for team communication** (not Slack)
13. **Retired government officials** as primary consultant pool
14. **Free assessment as lead magnet**: 10-15 questions, email capture, score generation

### MVP Document Handling Logic (from Dec 10 Meeting)

Three cases that must be handled in the property detail flow:

```
Case 1: No Documents
  → User gets free consultation offer
  → Can purchase document retrieval services
  → Assessment score: 0% (red)

Case 2: Partial Documents
  → User sees which docs are missing with guidance
  → Can select service to retrieve missing docs (system shows steps)
  → Assessment score: 20-79% (yellow)

Case 3: All/Most Documents Available
  → User can select verification services
  → Payment required before verification begins
  → Assessment score: 80-100% (green)
```

This logic should be implemented in the `PropertyCompletionService` (Phase 2, Stream A) and reflected in the property detail page UI (Phase 2, Stream B).

### Flow Logic Rules (for implementation)

#### 1. Property Type Selection
```
property_type ∈ {Land, Apartment, Other}
```
- Selection determines which fields appear next (future: JSON-driven per type)
- All three types share the same ownership and document flow for MVP

#### 2. Ownership Category Selection
```
ownership_category ∈ {Purchase, Gift, Succession, Court Decree, Settlement, Donation}
```
- This maps to the `ownership_statuses` table (6 active statuses)
- Determines which **conditional documents** are shown in the upload step:
  - Court Decree → show "Court Order / Decree" document type
  - Settlement → show "Settlement / Lease Gazettes" document type
  - Succession → show "Succession Certificate" document type
  - Purchase, Gift, Donation → no conditional documents

#### 3. Ownership Type Selection
```
ownership_type ∈ {Sole Ownership, Joint Ownership, Co-Ownership without Partition}
```
- **Sole Ownership**: No co-owner form; user is 100% owner
- **Joint Ownership**: Show co-owner form; each party has defined shares; total must = 100%
- **Co-Ownership without Partition**: Show co-owner form; property is undivided family property; ownership percentages represent legal share (important for inheritance cases)

#### 4. Document Upload and Scoring
```python
# Pseudocode for scoring
score = 0
for doc in uploaded_documents:
    score += document_types[doc.type_id].score_weight

if score == 0:
    service_path = "consultancy"        # Free consultation
elif score < 100:
    service_path = "retrieve_document"  # Help user get missing docs
else:
    service_path = "assessment"         # Full document assessment
    # Sub-options: verify_document OR list_missing_documents
```

#### 5. Service Path Determination
| Condition | Score | Service Offered | Description |
|-----------|-------|----------------|-------------|
| No documents | 0 pts | **Consultancy Service** | Free initial consultation; guide user on what to collect |
| Partial documents | 1-99 pts | **Retrieve Document** | Paid service to help user obtain missing documents on-ground |
| All documents | 100 pts | **Document Assessment** | Full verification and gap analysis of uploaded documents |

#### 6. Checkout Triggers
- **Consultancy Service (free)**: No Stripe checkout; create ticket directly
- **Retrieve Document (paid)**: Stripe checkout for document retrieval fee
- **Document Assessment (paid)**: Stripe checkout for assessment fee
- All paths lead to ticket creation + consultant assignment post-payment

### Conditional Document Display Logic

```typescript
// Frontend logic: which document types to show in upload step
function getDocumentTypes(ownershipStatusSlug: string): DocumentType[] {
  // Always show the 7 scored documents
  const scoredDocs = documentTypes.filter(d => d.score_weight > 0);

  // Add conditional documents based on ownership
  const conditionalDocs = documentTypes.filter(d =>
    d.conditional_on_ownership === ownershipStatusSlug
  );

  // Add any "additional documents" catch-all
  const additionalDocs = documentTypes.filter(d =>
    d.category === 'additional' && !d.conditional_on_ownership
  );

  return [...scoredDocs, ...conditionalDocs, ...additionalDocs];
}

// Example:
// ownershipStatus = "court_decree"
// → Shows: 7 scored docs + Court Order/Decree + Additional Documents
//
// ownershipStatus = "purchase"
// → Shows: 7 scored docs + Additional Documents (no conditional docs)
```
